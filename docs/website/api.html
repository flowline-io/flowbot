API Reference

Complete reference for Flowbot's REST API — endpoints, authentication, pagination, and error handling.

Base URL

http://localhost:6060

The listen address is configured in flowbot.yaml under the listen field (default :6060).

Authentication

All API requests must include an access token:

X-AccessToken: <your-token>

Tokens are managed through the admin CLI (flowbot-cli user token create). OAuth 2.0 is supported for select providers via /oauth/:provider/:flag.

Routing Planes

Flowbot divides its API into two distinct planes with separate prefixes:

Prefix Plane Purpose
/service/{capability}/* Capability Plane Business operations: bookmark, archive, reader, kanban, ...
/hub/* Management Plane App lifecycle, health, capability registry

Health Probes

Endpoint Method Auth Description
/livez GET No Liveness probe: returns 200 if process is running
/readyz GET No Readiness probe: checks DB, Redis, and dependencies
/startupz GET No Startup probe: returns 200 once init completes
/metrics GET No Prometheus-compatible metrics endpoint

Capability Plane

The capability plane provides a unified REST interface for all capabilities. Every capability supports the same standard operations regardless of the underlying provider.

Universal Operations

Operation Method Path Request
List GET /service/{capability} ?limit=20&cursor=opaque
Get GET /service/{capability}/{id} —
Create POST /service/{capability} JSON body
Update PUT /service/{capability}/{id} JSON body
Delete DELETE /service/{capability}/{id} —

Example: List Bookmarks

GET /service/bookmark?limit=20&cursor=eyJvZmZzZXQiOjIwfQ==
X-AccessToken: sk-abc123

# Response 200 OK
{
  "items": [
    {
      "id": "bkm_a1b2c3",
      "url": "https://example.com/article",
      "title": "An Interesting Article",
      "tags": ["homelab", "selfhosted"],
      "created_at": "2026-04-30T14:23:01Z"
    }
  ],
  "next_cursor": "eyJvZmZzZXQiOjQwfQ==",
  "has_more": true
}

Example: Create Bookmark

POST /service/bookmark
X-AccessToken: sk-abc123
Content-Type: application/json

{
  "url": "https://example.com/article",
  "title": "An Interesting Article",
  "tags": ["homelab", "selfhosted"]
}

# Response 201 Created
{
  "id": "bkm_a1b2c3",
  "url": "https://example.com/article",
  "title": "An Interesting Article",
  "tags": ["homelab", "selfhosted"],
  "created_at": "2026-04-30T14:23:01Z"
}

Pagination

All list endpoints use the same cursor-based pagination. Provider-specific page/offset/token internals are never exposed.

Request Parameters

Parameter Type Default Max Description
limit integer 20 100 Number of items to return
cursor string — — Opaque cursor for the next page

Response Fields

Field Type Description
items array Array of results for this page
next_cursor string | null Cursor to fetch the next page; null if no more
has_more boolean True if more pages are available
Cursor values are opaque. Clients must never parse or construct cursor values. Pass the next_cursor value as-is to the next request.

Hub (Management Plane)

Endpoint Method Auth Description
/hub/apps GET Yes List all registered applications
/hub/apps/:name GET Yes Get details for a specific app
/hub/apps/:name/restart POST Yes Restart a specific application
/hub/capabilities GET Yes List available capabilities and their bindings
/hub/health GET Yes Aggregated health status across all apps

Bot Module Routes

Each of the 20 bot modules can register additional webservice routes at runtime.

Prefix Module Description
/dev/* dev Debugging, testing, forms
/search/* search Full-text search: autocomplete, query
/user/* user User data: bookmarks, dashboard, kanban, metrics
/server/* server Server management: stack traces, uploads
/workflow/* workflow Workflow management and scheduling

Webhook & Callback Endpoints

Route Method Description
/webhook/:flag ALL Inbound webhook handler for external triggers
/chatbot/:platform ALL Chat platform callbacks (Discord, Slack, Tailchat)
/oauth/:provider/:flag ALL OAuth 2.0 callback endpoints
/form POST Form submissions from web/PWA
/agent POST Agent data endpoint
/p/:id GET PWA page rendering (go-app WASM)

Error Handling

All errors follow a consistent JSON format. HTTP status codes are used appropriately.

HTTP Status Codes

Status Code When
400 bad_request Invalid input or malformed body
401 unauthorized Missing or invalid access token
403 forbidden Insufficient permissions
404 not_found Resource does not exist
409 conflict Duplicate or version conflict
429 rate_limited Too many requests
500 internal_error Unexpected server error
503 service_unavailable Provider or dependency unavailable

Error Response Format

{
  "error": {
    "code": "not_found",
    "message": "Bookmark not found",
    "details": null
  }
}

Swagger / OpenAPI

Interactive API documentation is available at runtime:

http://localhost:6060/swagger/index.html

Regenerate the spec from code annotations:

go tool task swagger
