Technical Design

Architecture, data flows, design rules, and engineering decisions behind Flowbot.

Architecture Overview

Flowbot follows a strict six-layer architecture with a side management plane. Every layer has a single responsibility; no layer leaks into its neighbors.

Layer Stack

Layer Name Components
6 External Users, Discord/Slack/Tailchat, Third-Party APIs
5 Platform Chat platform protocol adapters
4 HTTP Gateway Fiber v3 server, REST API routing, auth middleware, rate limiting
3 Business Logic 20 bot modules, Workflow Engine, Pipeline Engine, LLM Agent
2 Capability ability.Invoke() — unified abstraction over all providers
1 Providers 17 service integrations: archivebox, karakeep, miniflux, kanboard, ...
0 Infrastructure MySQL (GORM), Redis (Streams), Docker (executor)

Management Plane (Side Plane)

Homelab Scanner → App Registry → Hub Manager → Capability Binding → Ability Layer

The Homelab Scanner discovers apps by scanning docker-compose.yaml files under /home/<user>/homelab/apps/. Discovered apps are registered in the Hub, then bound to capabilities (e.g., archivebox → archive).

Capability Abstraction Layer

The core of Flowbot's design. Modules never import providers directly — everything goes through ability.Invoke(). This decouples business logic from provider implementation details.

Standard Invocation Pattern

result, err := ability.Invoke(ctx, "bookmark", ability.OpList, ability.Params{
    Limit:  20,
    Cursor: "opaque_cursor_string",
})

Standard Errors

Provider-specific errors are wrapped into three standard error types:

Error Meaning
types.ErrNotFound Resource does not exist
types.ErrForbidden Access denied or auth failure
types.ErrProvider Provider-level error (wraps original)

Standard Pagination

All list operations use limit + opaque cursor. Provider-specific page/offset/token internals are hidden in adapters and never exposed to API callers or modules.

Declarative Pipeline Engine

Cross-service data flows are defined in YAML, triggered by durable events. Every pipeline run is persisted, idempotent, and fully audited.

Pipeline YAML Structure

trigger:
  event: "bookmark.created"          # or: cron, webhook, manual

steps:
  - action: archive.submit
    input: $.event.url
    idempotency_key: "archive-{{.event.id}}"

  - action: notify.send
    input:
      channel: slack
      message: "Archived: {{.steps.archive-submit.url}}"
    condition: "{{.steps.archive-submit.status}} == 'success'"

Durable Event Flow

DataEvent → MySQL (data_events table) → Redis Stream (outbox) → Pipeline Engine → Actions

Events are written to MySQL first for durability, then published to Redis Stream for real-time consumption. On restart, all unprocessed events are replayed from the database.

Pipeline Guarantees

  • Idempotency: Every step has a unique key; re-running is safe.
  • Atomicity: A pipeline completes all steps or records the failure point.
  • Auditability: Every run, step result, and error is written to the database.
  • Recoverability: Failed pipelines can be replayed from any step.

Workflow Engine

Composes capability invocations into a DAG (Directed Acyclic Graph). Each node is a step type; edges define execution order and data dependencies.

Step Types

Type Description
Capability Invokes a capability operation via ability.Invoke()
Message Sends a notification through the notify dispatcher
Fetch HTTP request to an external URL
Feed Parse and process RSS/Atom feeds
LLM Text generation, summarization, or tool use
Docker Execute a command inside a Docker container
Grep Pattern matching on previous step output
Unique Deduplicates items from a previous step
Torrent Torrent download management

AuthContext

A single AuthContext spans every interaction surface: REST, CLI, Chat, Form, Webhook, Cron, Pipeline, and Workflow. This ensures consistent identity propagation regardless of how a request enters the system. 

type AuthContext struct {
    UserID    string
    Roles     []string
    Platform  string   // "discord", "slack", "rest", "webhook", ...
    Interface string   // "chat", "cli", "web", ...
    Token     string
    SessionID string
}

Design Rules (Anti-Patterns)

The architecture enforces strict boundaries. Violating these is a design error:

  • Never import pkg/providers/* directly from internal/modules/* — use ability.Invoke()
  • Never call provider clients directly in modules
  • Never access local filesystem paths from modules
  • Never call Hub or Pipeline from inside a provider
  • Never emit DataEvent from inside a provider
  • Never write cross-service logic in cron/event handlers — use Pipeline
  • Never expose provider pagination internals to API callers
  • Never expose cursor/offset/page internal structure in API responses
  • Never use Redis Stream as sole long-term event store
